# ADR-004: Versioned Per-producer Snapshots Own Metric State

- **Status:** Accepted
- **Date:** 2026-07-23
- **Decision owners:** MetricShell maintainers
- **Related investigation:** `INV-004 — Metric-state Ownership and Semantics`
- **Evidence:** [INV-004](../../../research/INV-004/report.md)

## Context

MetricShell ingests application counters, gauges and histograms from one or more local producers. The ingestion model
must define which representation owns truth and remain deterministic after dropped, duplicate or reordered updates,
producer and receiver restarts, stale series, type conflicts and concurrent producers.

Four models were evaluated:

1. complete registry snapshots;
2. absolute values for individual series;
3. update operations such as increment, set and observe;
4. a hybrid model using operations as a fast path and complete snapshots for authority and reconciliation.

Operation-only state cannot recover a missed increment or an in-memory receiver restart. Unowned absolute counter
values permit decreases and make multi-producer aggregation ambiguous. Complete snapshots recover correctness but have
registry-size-dependent cost. The hybrid model retains fast-path efficiency while preserving a recoverable source of
truth.

## Decision

MetricShell uses **versioned complete per-producer snapshots as authoritative metric state**.

Sequenced operations are an optional acceleration path. They are not durable or authoritative by themselves. A final
state is valid only when every participating producer is complete according to the rules below.

### Ownership boundary

Every update is scoped by:

```text
(producer_id, producer_epoch, sequence)
```

- `producer_id` is stable for one logical producer identity;
- `producer_epoch` changes monotonically whenever that producer restarts or intentionally resets its cumulative state;
- `sequence` increases strictly within one producer epoch;
- older epochs and duplicate or older sequence numbers are rejected without mutating receiver state.

An operation carrying a previously unseen newer epoch does not establish authority. MetricShell marks that producer
epoch incomplete and requires its initial complete snapshot. Operations for the new epoch are accepted only after that
snapshot. This prevents a restarted producer with `seq=1` from being confused with an old update and prevents partial
new-epoch state from becoming final.

### Complete snapshot semantics

A complete snapshot replaces only the named producer's registry contribution.

- a series missing from a newer complete snapshot is removed for that producer;
- a newer snapshot repairs detected operation gaps and clears the producer's incomplete state;
- a receiver restart requires producers to republish authoritative snapshots;
- a final application state requires a final complete snapshot from each producer that used operations;
- a snapshot is rejected atomically when any included series violates its type, schema or monotonicity contract.

### Counters

Counter snapshot values are non-negative cumulative values.

A counter cannot decrease within one `(producer_id, producer_epoch)`. A new producer epoch may begin with a lower
counter value. Compatible per-producer counter contributions are summed for exposition.

### Gauges

A gauge uses an absolute value.

Two producers cannot export the same gauge series unless an explicit aggregation policy is configured. Supported
policies must be named and documented, such as `sum` or `max`; implicit last-writer-wins behavior is prohibited.

### Histograms

An authoritative histogram snapshot contains:

- ordered bucket boundaries;
- cumulative bucket counts;
- total observation count;
- observation sum.

The number and values of bucket boundaries are immutable within a producer epoch. Cumulative bucket counts and total
count cannot decrease within an epoch, and the final cumulative bucket equals `count`. Compatible histograms aggregate
component-wise: corresponding bucket counts, counts and sums are added. A boundary mismatch is rejected. A new epoch
may reset cumulative histogram state.

### Operation fast path

Operations carry the full ownership tuple and are applied transactionally.

- validation occurs before sequence state advances;
- a rejected type conflict does not consume its sequence;
- a sequence gap marks the producer incomplete;
- a duplicate does not create a gap;
- a late operation does not repair a gap;
- only a complete authoritative snapshot repairs incomplete state;
- operation-only state cannot be used as final state after a gap or receiver restart.

The reconciliation interval is configurable by a future transport/protocol decision. INV-004 does not select a
production default because snapshot cost depends on registry size, producer count, encoding and transport.

### Type and schema conflicts

A metric family type cannot change silently within an epoch or across simultaneous owner contributions.

Conflicting counter/gauge/histogram types, incompatible histogram boundaries and unconfigured duplicate gauges are
rejected. MetricShell must expose diagnostics or internal metrics for rejected updates and incomplete producers.

## Evidence

The decision is supported by two Docker/LinuxKit runs using the same benchmark-code fingerprint:

```text
e52784470ff33e35fb58ab142be26a345bb6a373bd2eac58666b269f56875fd6
```

The environments were:

- Docker Desktop on macOS/LinuxKit, `linux/aarch64`, Docker `29.4.3`;
- Ubuntu/LinuxKit, `linux/x86_64`, Docker `27.4.0`.

Across both environments:

- all 33 semantic scenarios produced their expected result;
- all 34 per-scenario/cardinality assertions passed;
- 29 accepted invariants and four deliberate counterexamples matched exactly;
- `semantics.tsv` and `assertions.tsv` were identical;
- dropped operations and operation-only receiver recovery failed as predicted;
- snapshots repaired loss, restart and stale-series state;
- old operation epochs were rejected;
- new operation epochs required an initial snapshot;
- type-conflict rejection did not consume sequence state;
- compatible histograms aggregated buckets/count/sum component-wise;
- histogram boundary mismatch and cumulative decreases were rejected;
- counter, gauge and multi-producer policies behaved consistently;
- 30-repetition snapshot, operation and amortized-hybrid benchmarks completed in each environment.

The benchmark measured throughput, allocation and reconciliation cost. It did not measure signal-to-exit latency,
because INV-004 does not start or signal a workload. Signal-to-exit evidence belongs to the lifecycle investigations.

Both tested container environments use LinuxKit. The results establish cross-architecture/container-environment
consistency for macOS/LinuxKit ARM64 and Ubuntu/LinuxKit x86-64, not native non-LinuxKit Linux or Kubernetes parity.

## Consequences

### Positive consequences

- Metric truth has one explicit owner and recovery path.
- Lost or reordered fast-path operations are detectable.
- Producer and receiver restarts cannot silently merge or discard epochs.
- Stale series have deterministic deletion semantics.
- Counters and histograms retain cumulative monotonicity within an epoch.
- Multiple producers can contribute compatible counters and histograms safely.
- Operations remain available for high-throughput ingestion between reconciliations.
- Final state can be classified as complete or incomplete rather than guessed.

### Negative consequences

- Hybrid clients and protocols are more complex than absolute-value or operation-only models.
- Producers must maintain stable identity, epoch and sequence metadata.
- Producers using operations must also publish initial, reconciliation and final snapshots.
- MetricShell must retain per-producer registries and completeness state.
- Complete snapshots amplify CPU, memory and transport cost with registry cardinality.
- Gauge aggregation requires explicit configuration.
- Diagnostics are required for gaps, rejected updates, incomplete owners and schema conflicts.

### Operational consequences

- Operators must treat producer identity changes as new ownership, not ordinary sequence resets.
- A producer that cannot provide an authoritative snapshot cannot use the operation fast path.
- Scraped final state must indicate or exclude incomplete producer contributions according to exposition policy.
- Reconciliation frequency must be tuned after transport benchmarks; one universal interval is not implied by this ADR.

## Rejected alternatives

### Complete global snapshots from an arbitrary producer

**Rejected.**

One producer cannot authoritatively delete or replace another producer's contribution. Snapshots are complete only
within their explicit producer ownership boundary.

### Absolute per-series values as the universal model

**Rejected.**

Absolute counters can decrease without an epoch transition, do not provide a registry boundary for stale deletion and
produce ambiguous last-writer-wins behavior with multiple producers.

Absolute set remains valid for gauges inside the accepted owner/epoch/sequence contract.

### Operations as the sole source of truth

**Rejected.**

A dropped increment permanently changes the result, and receiver restart loses in-memory state. Sequence gaps detect
the problem but cannot reconstruct the missing value. Operations therefore require snapshot reconciliation.

### Implicit last-writer-wins multi-producer gauges

**Rejected.**

Arrival order is not application semantics. Duplicate gauge ownership requires an explicit aggregation policy or is
rejected.

### Accept operations immediately after a producer epoch change

**Rejected.**

This would allow partial state from a restarted producer to become authoritative. A complete initial snapshot is the
required epoch-opening boundary.

## Decision boundaries

This ADR does **not** decide:

- the wire format for snapshots or operations;
- which ingestion transports support operations;
- the default reconciliation interval or operation batch size;
- durable receiver persistence or WAL design;
- final exposition behavior for incomplete state;
- cardinality and payload-size limits;
- authentication or producer authorization;
- Prometheus scrape-count and final-scrape semantics;
- native Linux, containerd, CRI-O or Kubernetes performance parity.

These questions belong to INV-005 through INV-011 and the security/resource-limit investigation.

## Implementation requirements

Production implementation derived from this ADR must include automated tests for:

- producer ID, epoch and sequence ordering;
- atomic rejection without sequence consumption;
- old-epoch rejection and new-epoch initial-snapshot gating;
- gap detection, duplicate handling and late-operation behavior;
- reconciliation and final snapshot completeness;
- stale-series deletion per producer;
- counter monotonicity and epoch reset;
- gauge collision rejection and each supported aggregation policy;
- histogram boundary/count/bucket validation;
- component-wise histogram aggregation;
- histogram monotonicity and epoch reset;
- cross-owner type conflicts;
- receiver restart and producer republish;
- diagnostics for rejected and incomplete state;
- bounded resource behavior under high series and producer cardinality.
