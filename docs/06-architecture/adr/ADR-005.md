# ADR-005: Stable Ingestion Transports Are File Snapshot, Unix Stream and Local HTTP

- **Status:** Accepted
- **Date:** 2026-07-23
- **Decision owners:** MetricShell maintainers
- **Related investigation:** `INV-005 — Ingestion Transport Comparison`
- **Evidence:** [INV-005](../../../research/INV-005/report.md)

## Context

MetricShell needs container-local mechanisms through which an application can publish metric state or updates. The
first stable release must balance PHP integration effort, delivery semantics, throughput, latency, recovery,
multi-producer behavior, protocol complexity, portability and resource control.

INV-005 compared:

1. file snapshot;
2. Unix domain stream socket;
3. Unix domain datagram socket;
4. loopback TCP HTTP;
5. local gRPC over a Unix socket;
6. POSIX shared-memory-backed mapping;
7. regular memory-mapped file.

The benchmark separates `publish-only`, `consumer-observed` and `acknowledged` contracts. Results from different
contracts are not treated as directly comparable.

## Decision

The first stable MetricShell release supports three ingestion transports:

- file snapshot;
- Unix domain stream socket;
- local HTTP over loopback TCP.

Unix datagram may be implemented as an optional adapter. It is not a stable core transport and must be described as
unacknowledged and potentially lossy.

gRPC, shared memory and mmap are excluded from the first stable transport set.

### File snapshot semantics

File ingestion represents replaceable latest state, not an event log.

The producer writes a new snapshot and atomically replaces the previous file within one filesystem. Concurrent
intermediate snapshots may be superseded before MetricShell observes them. Correctness is based on eventually observing
the latest valid state, not every replacement.

Detailed watching, reconciliation, invalid-file and directory-recreation behavior remains assigned to INV-006.

### Unix stream semantics

Unix stream is the stable event-oriented local transport.

The production protocol must provide framing, resource bounds and an explicit success/error acknowledgement when the
producer requires confirmed acceptance. MetricShell must detect disconnects and support producer reconnection.

Detailed framing, backpressure, timeout and credential behavior remains assigned to INV-007.

### Local HTTP semantics

Local HTTP is the stable request/response integration transport.

The baseline uses loopback TCP and an application response to confirm acceptance. The implementation must bind only to
the configured local interface by default and apply request-size, timeout and concurrency limits.

HTTP over a Unix socket is not required by this ADR. Detailed endpoint and payload behavior remains assigned to
INV-008.

### Optional Unix datagram semantics

Unix datagram has no acknowledgement guarantee in the selected model.

If provided, the adapter must:

- expose sent/observed/lost accounting where observable;
- document that successful producer `write` does not prove consumer acceptance;
- use an environment-appropriate bounded payload rather than a UDP/IPv4 constant;
- define behavior under socket-buffer saturation and consumer absence.

The measured AF_UNIX accepted boundary is evidence about the tested environments, not a portable protocol limit.

Across both environments:

- all 52 transport/profile/scenario cells were emitted;
- all runner assertions passed;
- all publish-only and acknowledged operations completed;
- Unix stream observed 100% of operations in all observed scenarios;
- file, shared memory and mmap observed 100% of single-producer publications;
- Unix datagram sent/observed/lost accounting was recorded for all four scenarios and observed 100% in these runs;
- four-producer file, shared-memory and mmap cells each recorded 6 superseded publications out of 2,000;
- all three executable PHP integrations passed for file, Unix stream and HTTP;
- missing Unix socket, refused HTTP, cross-filesystem rename, mmap resize and AF_UNIX datagram probes passed;
- the environment-specific AF_UNIX maximum accepted payload measured 212,960 bytes.

The two environments produced different performance distributions, but the correctness contracts and architectural
ranking were consistent. Full throughput and p50/p95/p99 values are retained in both `summary.tsv` files.

Signal-to-exit is not part of this decision because INV-005 does not supervise or signal a workload.

## Consequences

### Positive consequences

- The stable set covers snapshot, local stream and request/response integration styles.
- PHP delivery paths for all three stable candidates are executable.
- File snapshots provide a simple recovery baseline through replaceable state.
- Unix stream provides explicit framing and acknowledgement without HTTP parser overhead.
- HTTP provides familiar tooling and straightforward application integration.
- The default product avoids mandatory gRPC/protobuf and shared-memory protocol dependencies.
- Transport semantics are explicit rather than inferred from incomparable throughput numbers.

### Negative consequences

- MetricShell must maintain three stable ingestion front ends.
- File snapshot and event transports have different application semantics.
- Loopback HTTP requires port allocation and HTTP resource controls.
- Unix stream requires a production framing and backpressure protocol.
- Optional datagram users must accept and monitor loss.
- Shared-memory-class peak publication throughput is not available through the stable set.

### Operational consequences

- Operators choose file snapshot for latest-state publication.
- Producers requiring confirmed event acceptance use Unix stream or local HTTP.
- Datagram deployments must be explicitly enabled and documented as lossy.
- Local HTTP binds to loopback by default; it is not a remote ingestion API.
- Resource limits must be configurable per stable transport.

## Rejected alternatives

### Make Unix datagram a stable core transport

**Rejected.**

A successful datagram write does not confirm consumer observation. Socket-buffer behavior and accepted message size are
environment-dependent. Datagram remains an optional adapter for producers that accept loss.

### Include gRPC in the first stable release

**Rejected.**

The tested acknowledged gRPC transport did not provide an architectural capability absent from Unix stream or HTTP
that justifies the additional client, protocol and dependency complexity for the first release.

### Include shared memory or mmap in the first stable release

**Rejected.**

Low publication latency alone does not define a complete ingestion protocol. Notification, acknowledgement, memory
layout, synchronization, versioning, crash recovery and resizing remain unresolved. Concurrent publications can
supersede intermediate states.

### Select one transport for every producer semantic

**Rejected.**

Snapshot state, framed event delivery and request/response integration serve materially different producer needs.
Forcing one transport would either lose useful state semantics or add avoidable integration complexity.

## Decision boundaries

This ADR does **not** define:

- the file watcher/reconciliation algorithm;
- the production Unix stream wire format;
- HTTP payload representation and endpoint paths;
- metric-state update semantics beyond snapshot replacement;
- authentication beyond local binding and filesystem permissions;
- final-state scrape behavior;
- shared-memory/mmap feasibility for a future optional adapter;
- native non-LinuxKit performance or behavior.

## Implementation requirements

Production implementation derived from this ADR must include:

- file atomic-replacement and reconciliation tests;
- Unix stream framing, acknowledgement, reconnect, backpressure and size-limit tests;
- local HTTP request-size, timeout, concurrency and loopback-binding tests;
- executable PHP integration tests for every stable transport;
- structured accepted/rejected/error counters per transport;
- explicit latest-state semantics for file snapshots;
- no claim that a datagram write proves ingestion;
- bounded memory, file and connection resource usage;
- cross-architecture container integration tests.

## Limitations

Both evidence environments use LinuxKit. The matching aarch64/x86_64 results confirm cross-architecture behavior within
those container environments but do not verify native non-LinuxKit Linux.
