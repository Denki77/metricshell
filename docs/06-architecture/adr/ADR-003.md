# ADR-003: MetricShell Uses Explicit Shutdown Budgets and Deadline-Aware Finalization

- **Status:** Accepted
- **Date:** 2026-07-23
- **Decision owners:** MetricShell maintainers
- **Related investigation:** `INV-003 — Shutdown Time Budgeting`
- **Evidence:** [INV-003](../../../research/INV-003/report.md)

## Context

MetricShell owns one workload execution and must preserve its outcome while shutting down inside an external container
runtime or orchestrator grace period. After receiving termination, MetricShell may need to:

- stop admitting new HTTP scrapes;
- forward the termination signal to the workload process group;
- wait for cooperative workload shutdown;
- force-kill a non-cooperative workload;
- observe and preserve the workload outcome;
- finalize application metric state and diagnostics;
- drain already-active HTTP requests;
- exit before an external SIGKILL.

Giving the entire external grace period to the workload leaves no deterministic time for MetricShell. Reserving an
unbounded or percentage-only interval makes behavior difficult to reason about across short and long grace periods.
MetricShell also cannot infer a Docker stop timeout from the received signal.

INV-003 compared:

1. a fixed MetricShell reserve subtracted from total grace;
2. a percentage workload share;
3. explicit workload timeout plus explicit MetricShell reserve;
4. an externally supplied absolute deadline with remaining time computed when termination begins.

## Decision

MetricShell uses an **explicit workload shutdown timeout and an explicit MetricShell shutdown reserve**, validated
against a known total grace period or an externally supplied absolute deadline.

An absolute deadline, when available, is the authoritative upper bound. MetricShell computes remaining time when
termination begins and caps every subsequent phase by that remaining time.

### Budget validation

Configuration is rejected when:

- total grace is zero or negative;
- a phase duration is negative;
- a configured ratio is outside `[0,1]`;
- `workload_timeout + shutdown_reserve > total_grace`.

MetricShell does not silently overcommit or expand an external deadline.

When an absolute deadline is already expired, or the configured reserve is greater than remaining time, workload grace
is zero. MetricShell immediately proceeds with bounded termination and finalization.

### Workload shutdown

On TERM or INT, MetricShell:

1. records the shutdown start and resolves the effective deadline;
2. stops admitting new HTTP requests;
3. forwards the signal to the workload process or process group;
4. waits no longer than the resolved workload budget;
5. sends SIGKILL when that budget expires;
6. preserves the resulting workload exit semantics.

A workload killed by MetricShell after its budget expires resolves to signal-derived exit `137`.

### MetricShell reserve

The reserve belongs to MetricShell, not the workload. It covers workload result observation, metric/diagnostic
finalization and bounded HTTP drain.

The researched starting values are:

- 1 s external total: 750 ms workload budget and 250 ms reserve;
- 5–10 s external total: reserve at least 1 s;
- 30–60 s external total: reserve 2 s.

The 250 ms reserve is the minimum tested short-window value, not a universal production default. Deployments with CPU
throttling, slow storage, large metric state or high HTTP concurrency must size the reserve from representative
measurements.

### HTTP shutdown

MetricShell closes the HTTP listener when termination begins.

Requests already active may drain, bounded by both an HTTP phase timeout and the absolute remaining deadline. Requests
arriving after shutdown starts are not admitted as normal scrapes.

MetricShell does not begin a configurable post-exit scrape-wait interval after external termination has started.
Post-exit waiting remains available only after natural workload completion and must still be finite.

### Deadline input

MetricShell supports a known total grace period as the portable baseline.

Where a runtime or control plane can supply an absolute deadline, MetricShell accepts it and calculates remaining time
at shutdown. The investigation prototype used an absolute Unix-millisecond value supplied directly or through a file
read when TERM arrived.

The production transport for supplying an absolute deadline may differ, but it must preserve the same semantics:
deadline resolution occurs at shutdown, and phase budgets use actual remaining time.

### Derived policies

A fixed reserve is acceptable as a deployment default after it is derived from measured MetricShell finalization and
drain requirements.

A percentage-only reserve is not accepted as the sole production policy because it provides too little absolute margin
for short windows and unnecessarily withholds workload time in long windows.

## Evidence

The accepted benchmark fingerprint is:

```text
27e8a991546667f92abb5965c044b834c1583f710cbb060dfef81174b29bb53c
```

Matching-fingerprint runs were collected in:

- Docker Desktop on macOS/LinuxKit, `linux/aarch64`, Docker 29.4.3;
- Docker on Ubuntu/LinuxKit, `linux/x86_64`, Docker 27.4.0.

Across both environments:

- all 20 shutdown-grid cases passed;
- all 210 explicit grid assertions passed;
- all 30 repeated TERM-to-shutdown samples passed;
- all five absolute-deadline scenarios passed;
- both HTTP admission/drain scenarios passed;
- both independent Docker deadline scenarios passed;
- explicit budget overflow was rejected;
- forced workloads exited `137`;
- cooperative workloads exited `0`;
- each grid case recorded exactly one shutdown, finalization and HTTP-shutdown marker;
- no forced workload was killed before its workload budget;
- no normal post-TERM scrape was admitted.

TERM-to-complete-shutdown p50/p95/p99 was:

- macOS/LinuxKit aarch64: `6.825/7.710/7.986 ms`;
- Ubuntu/LinuxKit x86_64: `7.767/8.682/8.941 ms`.

For independent Docker deadlines, MetricShell used internal totals smaller than the external windows:

- 750 ms internal within 1 s external: completion `527.823 ms` on macOS and `525.371 ms` on Ubuntu;
- 4 s internal within 5 s external: completion `3529.925 ms` on macOS and `3523.575 ms` on Ubuntu.

Both container environments use LinuxKit. The evidence covers two host environments and CPU architectures, but does not
establish native non-LinuxKit Linux, containerd, CRI-O or Kubernetes parity.

## Consequences

### Positive consequences

- Shutdown behavior has explicit phase ownership and deterministic upper bounds.
- Workload time cannot accidentally consume MetricShell's entire external grace period.
- Invalid overcommitted configurations fail visibly.
- Absolute deadlines account for grace already consumed before MetricShell receives TERM.
- New HTTP work stops immediately while active work remains bounded.
- Forced workload exit semantics remain observable.
- The policy behaved consistently in matching-fingerprint aarch64 and x86_64 LinuxKit environments.

### Negative consequences

- Operators must keep MetricShell's total grace consistent with Docker, Compose or orchestrator configuration when no
  absolute deadline is available.
- Reserve sizing depends on real finalization cost, resource pressure and HTTP concurrency.
- Supplying an absolute deadline requires a runtime or control-plane integration.
- Short total windows leave little operational margin.
- Deadline and phase-timer implementation introduces additional shutdown race conditions that require integration tests.

### Operational consequences

- Docker `stop --time`, Compose `stop_grace_period` and MetricShell total grace must agree.
- Diagnostics must expose configured total, resolved remaining time, workload budget, reserve, forced-kill state,
  finalization duration, HTTP drain duration and total shutdown duration.
- Deployments should benchmark reserve sizing under representative CPU, memory, storage and scrape load.
- Post-exit scrape waiting must not be used as part of signal-driven termination.

## Rejected alternatives

### Give the full external grace period to the workload

**Rejected.** MetricShell would have no deterministic time for finalization, active-request drain or orderly exit and
could be externally killed before recording completion.

### Use only a percentage reserve

**Rejected.** The absolute safety margin varies with total grace and is not stable across the tested 1–60 second range.

### Silently clamp an overcommitted explicit configuration

**Rejected.** Silent clamping changes the operator's workload contract and hides unsafe deployment configuration.

### Wait for new post-exit scrapes after TERM

**Rejected.** New waiting competes directly with the external deadline. MetricShell drains only requests active when
termination begins.

### Infer the Docker stop timeout from SIGTERM

**Rejected.** Docker does not include the stop timeout or absolute deadline in the signal.

## Decision boundaries

This ADR does not decide:

- the production transport used by Kubernetes or another orchestrator to provide an absolute deadline;
- final metric scrape acknowledgement or scrape-count semantics;
- metric ingestion transport or registry implementation;
- the cost of production metric-state serialization and durable diagnostics;
- reserve values under CPU throttling, memory pressure, slow storage or very high HTTP concurrency;
- behavior on native non-LinuxKit Linux, containerd, CRI-O or Kubernetes.

## Implementation requirements

Production implementation must test:

- explicit total, workload timeout and reserve validation;
- fixed-reserve and explicit-budget calculations;
- absolute deadline with full, partially consumed, nearly expired and expired remaining time;
- reserve greater than remaining time;
- cooperative workload shutdown;
- workload timeout and forced SIGKILL;
- preservation of exit `0`, arbitrary non-zero outcomes and signal-derived `137`;
- exactly one shutdown, finalization and HTTP-shutdown completion marker;
- no forced kill before workload budget expires;
- listener closure or documented rejection for requests arriving after shutdown begins;
- bounded drain for requests active before shutdown;
- finalization and HTTP phase caps against the absolute remaining deadline;
- external runtime deadline greater than the internal total with a measurable safety margin;
- no post-exit waiting after external termination begins;
- matching behavior on supported CPU architectures and container runtimes.
