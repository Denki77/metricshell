# ADR-002: MetricShell Executes One Workload and Delegates Restarts

- **Status:** Accepted
- **Date:** 2026-07-21
- **Decision owners:** MetricShell maintainers
- **Related investigation:** `INV-002 — Workload Lifecycle and Exit Semantics`
- **Evidence:** [INV-002](../../../research/INV-002/report.md)

## Context

MetricShell starts an application workload, owns its metric state, exposes metrics while it runs, may retain final state
for a bounded interval, and terminates with a resolved outcome. The lifecycle must define whether MetricShell executes
the workload once or restarts it internally.

Internal restart creates ambiguity for application counters. Resetting state between attempts makes a counter decrease
within one MetricShell process, while preserving state merges independent application executions. It also duplicates
retry count, eligibility and backoff already implemented by Docker, Compose and Kubernetes.

## Decision

MetricShell executes exactly one workload per MetricShell process.

MetricShell does not restart the workload. Docker, Compose or the orchestrator owns retry count, restart eligibility,
delay and backoff. Every new MetricShell process creates a new application metric-state epoch.

The required lifecycle is:

```text
initialize
  -> prepare ingestion and exposition
  -> start one workload
  -> run and expose one metric-state epoch
  -> observe exit, signal termination, OOM outcome, or start failure
  -> finalize and freeze metric state
  -> optionally expose final state for a finite configured interval
  -> exit once with the resolved outcome
```

There is no transition from workload exit back to workload start within one MetricShell process.

### Exit semantics

- workload exit `0` produces MetricShell exit `0`;
- workload exit `N` preserves `N`;
- signal termination `S` produces `128 + S`;
- workload start failure is distinct, with `127` used by the prototype;
- MetricShell internal failure uses a documented MetricShell-owned outcome;
- successful post-exit exposition does not replace the already resolved workload outcome.

### Post-exit availability

Post-exit availability is optional and finite. `0` is valid. INV-002 verifies functional values from `0` through `30s`
but does not select a production default; shutdown-budget allocation remains assigned to INV-003.

The implementation must measure its bounded interval inside MetricShell. Host-side `docker run` or `docker wait` time
is not an acceptable timer assertion because Docker CLI and daemon transport latency varies by environment.

### Restart and scrape semantics

An external restart creates two distinct MetricShell lifecycles and two metric epochs. Correctness requires the expected
process outcomes, exactly one workload execution per MetricShell process, and an externally visible restart count.

Whether a scrape observes the first epoch, the second epoch, or a restart gap is timing-dependent evidence rather than a
portable pass criterion. This ADR does not guarantee scrape continuity or a final scrape across restart.

## Evidence

The corrected reference benchmark fingerprint is:

```text
a8042a6b12b8d659f701125584223373a934186d45fdeae2e64f89f6362c05f2
```

The decision is confirmed by matching-fingerprint Docker Desktop/LinuxKit runs on macOS `linux/aarch64` and Ubuntu
`linux/x86_64`:

- all 8 core cases and all 6 core assertions passed in both environments;
- all 30 repeated single-execution cases passed in both environments;
- all extended assertions and result checks passed in both environments;
- signal-to-exit p50/p95/p99 was `0.594/1.194/2.010 ms` on macOS and `1.767/2.137/2.187 ms` on Ubuntu, with 30/30 passing samples in each;
- Docker and Compose each completed two expected single-workload lifecycles with restart count `1`;
- TERM, KILL and container-OOM scenarios produced `143`, `137` and `137`;
- post-exit settings `0, 1, 2, 5, 10, 30s` passed using the MetricShell internal timer;
- internal retry demonstrated counter reset `5 -> 2` and merged state `5 -> 7`;
- 10/100/1000 internal-attempt storms demonstrated extra supervisor work and linear log growth.

Ubuntu used Docker 27.4.0, LinuxKit 6.10.14 and x86_64; macOS used Docker 29.4.3, LinuxKit 6.12.76 and aarch64. The same
benchmark fingerprint was recorded in all four core/extended reference runs.

Both available container environments use LinuxKit. The evidence therefore covers two host environments and CPU
architectures, but not a native non-LinuxKit Linux kernel.

## Consequences

### Positive consequences

- One MetricShell lifetime has one deterministic workload outcome.
- One process lifetime has one unambiguous application metric epoch.
- Retry count and backoff remain visible in deployment configuration.
- MetricShell does not become a second orchestration controller.
- Docker, Compose and Kubernetes can apply their native restart policies.
- Counter reset semantics match normal process restarts.

### Negative consequences

- Metric state is not continuous across external restarts.
- Prometheus may miss a final epoch or observe a gap around restart.
- Operators must configure retry limits and backoff outside MetricShell.
- Final-state reliability needs a separate design and cannot assume a successful scrape.

### Operational consequences

- Do not expose an internal workload-retry option in production configuration.
- Document that every container restart creates a new metric epoch.
- Record workload outcome, lifecycle identity and restart diagnostics.
- Keep post-exit duration finite and within the shutdown budget decided by INV-003.
- Test runtime restart policy in Docker, Compose and Kubernetes deployment examples.

## Rejected alternatives

### Restart internally and reset metric state

**Rejected.** A counter decreased from `5` to `2` inside one MetricShell process, violating simple counter expectations
and making lifecycle identity implicit.

### Restart internally and preserve metric state

**Rejected.** A counter changed from `5` to `7` by merging two independent application executions into one epoch.

### Configure retries in both MetricShell and the runtime

**Rejected.** Nested retry controllers multiply attempts, obscure the effective limit and introduce competing backoff
and cancellation behavior.

## Decision boundaries

This ADR does not decide:

- production post-exit duration or shutdown reserve;
- final scrape counting or acknowledgement rules;
- metric ingestion transport;
- Kubernetes Job/CronJob discovery behavior;
- the permanent registry of MetricShell-owned internal exit codes;
- native non-LinuxKit Linux support evidence.

## Implementation requirements

Production implementation must test:

- exactly one workload start per MetricShell process;
- exit `0`, arbitrary non-zero exit, TERM, KILL and start failure;
- externally restarted processes each executing once;
- a new metric epoch after restart;
- internal post-exit timing independent of host CLI latency and locale;
- post-exit endpoint availability with preserved result;
- no internal retry configuration;
- timing-dependent scrape visibility as observation, not a portable invariant.
