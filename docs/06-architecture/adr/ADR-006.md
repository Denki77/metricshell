# ADR-006: File Ingestion Uses Directory inotify with Periodic Reconciliation

- **Status:** Accepted
- **Date:** 2026-07-23
- **Decision owners:** MetricShell maintainers
- **Related investigation:** `INV-006 — File-Based Ingestion`
- **Evidence:** [INV-006](../../../research/INV-006/report.md)

## Context

MetricShell supports file snapshot as a stable ingestion transport. The file represents replaceable complete metric
state, not an append-only event log. MetricShell must detect updates safely inside a Linux container while tolerating:

- a file that exists before MetricShell starts;
- a file that is created later;
- same-directory temporary-file writes followed by atomic rename;
- repeated replacement and coalesced intermediate versions;
- a writer that crashes before rename;
- invalid content;
- file deletion;
- directory deletion and recreation;
- inotify queue overflow or lost events;
- MetricShell restart.

INV-006 compared fixed-interval polling, directory-level inotify, and directory-level inotify combined with periodic
reconciliation. It measured writable container layer, container-local tmpfs and Docker named volume behavior.

## Decision

MetricShell file ingestion uses **directory-level inotify plus periodic reconciliation**.

### Watch scope

MetricShell watches the directory containing the configured snapshot, not only the snapshot inode.

This is required because the producer replaces the file through rename. A file-level watch follows the old inode and
does not reliably represent later replacements.

MetricShell also retains enough parent-directory visibility to recover when the watched directory is removed and
recreated.

### Reconciliation

MetricShell reconciles the current file:

- once during startup;
- immediately after a relevant inotify event;
- immediately after `IN_Q_OVERFLOW`, watch invalidation or watch reinstallation;
- periodically every `1s` by default.

The tested acceptable reconciliation range is `100ms–1s`. The production default is `1s`: normal updates still use
the event path, while the slower periodic interval reduces idle reads and CPU.

The reconciliation interval must be configurable and finite. Disabling periodic reconciliation is not a supported
correctness mode.

### Snapshot contract

The file contains complete registry state.

The producer must:

1. create a temporary file in the destination directory;
2. write one complete snapshot;
3. close the temporary file;
4. atomically rename it over the configured snapshot.

MetricShell may coalesce or skip intermediate snapshots. It guarantees convergence to the latest valid snapshot, not
delivery of every replacement.

### Validation and state replacement

MetricShell parses and validates a candidate snapshot before replacing the active metric state.

Invalid or incomplete content does not erase or partially mutate the last valid state. File deletion, a writer crash
before rename and an invalid replacement retain the last valid state while diagnostics record the condition.

A successful reload must be based on validated content or an explicit producer version, not only mtime.

### Overflow and event loss

`IN_Q_OVERFLOW` is treated as evidence that event history is incomplete. MetricShell immediately reconciles complete
state and reinstalls watches where necessary.

Real queue overflow occurrence is environment-dependent and is not a portable success criterion. Portable correctness
requires final-state convergence under overflow pressure.

Periodic reconciliation remains mandatory even though an observed overflow can trigger immediate recovery. A lost
event may leave no subsequent notification. Controlled INV-006 tests demonstrated that inotify-only did not recover in
this case, while the hybrid model recovered on its reconciliation timer.

### Storage scope

The primary supported storage locations are:

- the writable container layer;
- container-local tmpfs.

A Docker named volume follows the same tested behavior but remains informational for the first implementation.

Host bind mounts and their host filesystem event semantics are outside the primary support contract.

### Initial resource bounds

The prototype tested snapshots from `128 B` through `1 MiB`. This is a feasibility range, not the production maximum.

Production code must configure and enforce:

- maximum snapshot bytes;
- maximum parsed series count;
- maximum label count and label bytes;
- parser time and allocation bounds;
- bounded diagnostic cardinality.

Final limits require benchmarks with the selected production format and parser.

## Evidence

Two Docker Desktop/LinuxKit result sets used the identical benchmark fingerprint:

```text
4238d6e1be961e2d864ccfc95c763c1c9a03365c74d9c868e38f1b0d96eb1580
```

| Environment           | Docker | Architecture | Kernel           | Result set         |
|-----------------------|-------:|--------------|------------------|--------------------|
| macOS Docker Desktop  | 29.4.3 | aarch64      | LinuxKit 6.12.76 | `20260723T160155Z` |
| Ubuntu Docker Desktop | 27.4.0 | x86_64       | LinuxKit 6.10.14 | `20260723T161216Z` |

Across both environments:

- all 162 correctness assertions passed;
- all six controlled lost-event A/B assertions passed;
- all 12 10,000-update burst cases converged to the final snapshot;
- all six 20,000-update overflow-pressure cases converged to the final snapshot;
- real `IN_Q_OVERFLOW` was observed in four of six pressure cases;
- inotify-only failed to recover from a controlled event loss without a later event on all three filesystems;
- hybrid recovered from the same controlled loss on its reconciliation interval;
- every hybrid paced performance row completed without a missed update;
- invalid, partial and deleted files retained the last valid state;
- directory recreation and MetricShell restart recovered current valid state.

For 4 KiB snapshots, mean-of-three p95 detection latency across both environments was:

| Candidate                | Observed p95 range |
|--------------------------|-------------------:|
| Poll 10 ms               |   10.709–11.786 ms |
| Poll 100 ms              | 100.778–103.679 ms |
| inotify                  |     0.604–1.202 ms |
| Hybrid, reconcile 100 ms |     0.489–0.988 ms |
| Hybrid, reconcile 1 s    |     0.646–1.083 ms |

Five-second idle CPU for hybrid with 1 s reconciliation was 0.272–0.418% on macOS/LinuxKit and 0.402–0.438% on
Ubuntu/LinuxKit. Polling every 10 ms measured 2.593–3.014% and 4.496–5.085%, respectively.

The environments produced different timing distributions, but portable correctness, recovery behavior and candidate
ranking were consistent.

Signal-to-exit is not part of this decision because INV-006 does not supervise or signal a workload. Its corresponding
latency measure is producer timestamp to observed file update.

## Consequences

### Positive consequences

- Normal updates have event-driven latency.
- Periodic reconciliation provides bounded recovery after silent event loss.
- Atomic replacement prevents MetricShell from reading producer partial writes as committed snapshots.
- Complete-state semantics make coalescing and overflow recovery deterministic.
- Last-valid-state retention avoids exposing partially parsed or invalid registry state.
- Startup, directory recreation and MetricShell restart use the same reconciliation path.
- The design works consistently across the tested aarch64 and x86_64 LinuxKit environments.

### Negative consequences

- File ingestion requires Linux-specific inotify implementation.
- MetricShell must maintain watch-reinstallation and reconciliation state.
- A periodic scan adds steady-state reads and CPU even when no file changes.
- Complete snapshots can become expensive as file size and series count grow.
- Intermediate snapshot versions are not guaranteed to be observed.
- Last-valid-state retention requires explicit stale/error diagnostics so operators can detect a broken producer.

### Operational consequences

- Producers must use same-directory atomic rename.
- Operators should place high-churn ephemeral snapshots on container-local tmpfs when appropriate.
- Operators must not assume host bind mounts have the same event semantics.
- Metrics and logs must expose reload success, validation failure, watch invalidation, overflow and reconciliation.
- The configured reconciliation interval defines the recovery bound after a silent missed event.

## Rejected alternatives

### Polling only

**Rejected as the default.**

Polling is simple and recovered state correctly, but update latency follows the interval. Polling every 10 ms consumed
materially more idle CPU, while 100 ms and 1 s polling delayed normal detection.

Polling remains the conceptual recovery mechanism used by reconciliation.

### inotify only

**Rejected.**

Inotify provides fast normal detection and can immediately reconcile after a reported overflow or invalidation, but it
does not provide bounded recovery when an event is silently lost and no later event arrives.

The controlled lost-event A/B test demonstrated this boundary in both environments.

### Producer-triggered reload

**Rejected as a requirement for file ingestion.**

A second signal, socket or HTTP mechanism adds another transport and still does not remove the need to reconcile after
MetricShell restart, producer crash or lost notification.

Producer-triggered hints may be added later as an optimization but cannot be the sole correctness mechanism.

### Watch only the file inode

**Rejected.**

Atomic replacement changes the inode. Directory watching is required to observe the supported producer contract.

## Decision boundaries

This ADR does **not** define:

- the production snapshot serialization format;
- final maximum snapshot or series size;
- remote or host-mounted filesystem support;
- power-loss durability or fsync policy;
- append-only event delivery;
- exactly-once observation of intermediate versions;
- native non-LinuxKit Linux behavior;
- Kubernetes emptyDir behavior;
- metric exposition or final-scrape semantics.

## Implementation requirements

Production implementation derived from this ADR must include:

- Linux-specific directory inotify support;
- initial reconciliation before serving file-derived state;
- configurable periodic reconciliation with a `1s` default;
- immediate reconciliation after relevant events, overflow and watch invalidation;
- watch reinstallation after directory replacement;
- same-directory atomic-rename integration tests;
- last-valid-state retention tests for invalid, partial and deleted inputs;
- controlled silent-event-loss tests proving bounded hybrid recovery;
- overflow-pressure tests based on final-state convergence;
- startup and restart recovery tests;
- explicit snapshot byte, parser and series limits;
- reload/error/overflow/reconciliation diagnostics;
- cross-architecture container tests.

The research runner and committed evidence are architecture evidence. Production code requires independent unit,
integration and end-to-end coverage.

## Limitations

Both evidence environments use LinuxKit. Matching aarch64/x86_64 results confirm cross-architecture behavior inside
those container environments but do not verify native non-LinuxKit Linux.

The prototype uses a synthetic record and SHA-256 validation. It does not model the final parser, registry allocation,
series cardinality or power-loss durability.
