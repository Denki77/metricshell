# ADR-001: MetricShell Owns the Workload Lifecycle and Runs as PID 1

- **Status:** Accepted
- **Date:** 2026-07-19
- **Decision owners:** MetricShell maintainers
- **Related investigation:** `INV-001 — Process and PID 1 Model`
- **Evidence:** [INV-001](../../../research/INV-001/report.md)

## Context

MetricShell is a container runtime wrapper that starts one workload, exposes metrics for that workload, preserves its
result, and may remain alive for a bounded period after the workload has exited.

This role requires MetricShell to control the workload lifecycle rather than merely launch a child process. It must:

- start the workload;
- receive and forward operating-system signals;
- observe and preserve workload exit semantics;
- manage the workload process tree where responsibility belongs to MetricShell;
- reap adopted descendants;
- terminate non-cooperative workloads after a bounded grace period;
- remain available for bounded post-workload metric exposition when configured;
- exit with the previously resolved workload result unless MetricShell itself fails.

The process model is especially important inside containers because the process running as PID 1 has special
responsibilities and signal behavior. The investigation compared three broad candidates:

1. MetricShell runs directly as PID 1 and implements the required init and supervisor behavior.
2. Tini or Docker `--init` runs as PID 1, MetricShell runs below it, and MetricShell still supervises the workload.
3. Another external supervisor such as dumb-init, s6, or supervisord owns process management.

The decision must minimize operational dependencies while preserving deterministic lifecycle and exit behavior.

## Decision

MetricShell **runs as PID 1 and owns the lifecycle of exactly one workload execution**.

MetricShell is the workload supervisor even when an external init process is placed above it.

The runtime process model has the following required properties.

### Workload ownership

MetricShell starts exactly one workload as its direct child.

MetricShell does not restart the workload. Restart policy belongs to the container runtime or orchestrator.

MetricShell waits for the workload and resolves the workload result before it exits or enters a configured bounded
post-exit state.

### Signal handling

MetricShell explicitly handles and forwards supported termination and control signals.

At minimum, the process model must support forwarding:

- `SIGTERM`;
- `SIGINT`;
- `SIGHUP`;
- `SIGQUIT`.

Signals are forwarded either to the direct workload process or to the workload process group, according to the selected
process-group policy.

MetricShell must not rely on default PID 1 signal behavior.

### Process groups

MetricShell can create a dedicated process group for the workload.

A dedicated process group is required when the configured behavior expects signals to reach the workload and its
descendants rather than only the direct child.

Whether process-group mode is always enabled by default is not decided by this ADR. The final default belongs to the
detailed lifecycle and compatibility design.

### Descendant reaping

When MetricShell is PID 1, it is responsible for reaping adopted orphan descendants.

When MetricShell runs below an external init process, the external init normally owns descendants reparented to PID 1.
If MetricShell must retain visibility and cleanup responsibility for daemonized descendants in this mode, MetricShell
must enable Linux `PR_SET_CHILD_SUBREAPER`.

Daemonized descendants are not treated as independent workloads and do not change the result of the primary workload.

### Exit semantics

MetricShell preserves the resolved workload outcome.

The baseline mapping is:

- workload exits with `0` → MetricShell exits with `0`;
- workload exits with non-zero code `N` → MetricShell exits with `N`;
- workload terminates from signal `S` → MetricShell exits with `128 + S`;
- workload cannot be started → MetricShell exits with `127`;
- MetricShell fails internally before resolving a workload result → MetricShell uses a MetricShell-owned internal error
  code.

The prototype used exit code `70` for an internal failure. The permanent internal error-code registry may be refined
separately, but internal failures must remain distinguishable from workload results.

### Forced termination

After forwarding a termination signal, MetricShell waits only for a bounded workload grace period.

If the workload does not terminate within the configured period, MetricShell force-terminates the responsible workload
process or process group and preserves signal-based exit semantics.

This ADR requires a bounded force-kill fallback but does not define the production timeout value or the allocation of
the
external shutdown budget.

### Post-workload availability

MetricShell may continue exposing metrics for a bounded period after the workload has exited.

Entering this state must not overwrite or lose the already resolved workload result.

When the bounded post-exit condition completes or expires, MetricShell exits with the preserved workload result unless a
later MetricShell-owned failure takes precedence according to a separately defined failure policy.

### Shell-form workloads

Shell-form commands and shell wrapper scripts are supported with documented limitations.

A shell wrapper and its descendants may receive `SIGTERM` correctly while the shell still resolves to exit code `143`.
This is valid signal-exit behavior and is not equivalent to a clean application-controlled exit.

Users requiring precise application shutdown semantics should prefer exec-form commands without an unnecessary shell
wrapper.

## Evidence

The decision is supported by the reproducible `INV-001` prototype and benchmark runner.

Two reference runs used the same benchmark-code fingerprint:

```text
35dc9c63a0a9f6dedf56a1c6c80b582919d5961b8f233c49ef1aed55652b71fb
```

The environments were:

- Docker Desktop on macOS/LinuxKit, `linux/aarch64`;
- Docker Desktop on Ubuntu/LinuxKit, `linux/x86_64`.

Across both environments:

- all 52 benchmark cases passed;
- all 141 case-level assertions passed;
- direct-child signal receipt was confirmed;
- shell-parent and grandchild signal receipt was confirmed;
- workload exit codes and signal exit codes were preserved;
- PID 1 descendant reaping was confirmed;
- behavior under Docker init without subreaper was confirmed;
- behavior under Docker init with `PR_SET_CHILD_SUBREAPER` was confirmed;
- forced termination of a non-cooperative workload was confirmed;
- post-workload `/metrics` availability preserved the original workload result;
- no persistent zombies were observed in the sampled child-churn scenarios.

The benchmark establishes architecture feasibility and cross-environment behavioral consistency. It is not a production
performance guarantee.

## Consequences

### Positive consequences

- MetricShell has deterministic ownership of the behavior needed for metric finalization and post-exit exposition.
- The default image requires only the MetricShell binary and the workload.
- Workload exit status remains observable and reproducible.
- Signal handling is explicit rather than dependent on implicit PID 1 behavior.
- External init support remains possible without making it mandatory.
- The process model works consistently across the tested ARM64 and x86-64 Linux container environments.
- Restart responsibility remains outside MetricShell, avoiding a second orchestration layer.

### Negative consequences

- MetricShell must implement and maintain correct supervisor behavior.
- Signal forwarding, process groups, child waiting, descendant reaping, and shutdown races require extensive integration
  tests.
- Shell wrappers can obscure application-level exit semantics.
- External-init mode introduces an additional process layer and conditional subreaper behavior.
- Linux-specific functionality such as `PR_SET_CHILD_SUBREAPER` requires platform-specific implementation and explicit
  fallback behavior on unsupported systems.
- MetricShell must reserve enough shutdown time for its own finalization after workload termination.

### Operational consequences

- Container examples should use MetricShell as the image entrypoint or top-level command.
- Documentation should prefer exec-form workload arguments.
- Kubernetes, Docker, and Docker Compose restart policies remain the operator's responsibility.
- Users enabling an external init process must understand whether descendant ownership belongs to the init process or
  must be retained by MetricShell through subreaper mode.
- Diagnostics must expose enough process-model information to identify PID, process-group, subreaper, signal, and exit
  behavior.

## Rejected alternatives

### Require Tini as PID 1

**Rejected.**

Tini can provide namespace-init behavior, but it does not remove MetricShell's need to own workload startup, signal
forwarding, result preservation, shutdown policy, and post-exit behavior.

It also adds another binary, process layer, configuration mode, and descendant-ownership distinction.

Requiring Tini as part of the default runtime architecture was rejected.

### Delegate process management to another supervisor

**Rejected.**

A general supervisor such as s6 or supervisord adds operational scope and configuration that MetricShell does not need.
MetricShell supervises one workload, not a set of independently managed services.

Delegating lifecycle ownership would also make workload result preservation and post-exit metric behavior dependent on
external supervisor semantics.

### Run MetricShell beside the workload without lifecycle ownership

**Rejected.**

A side-by-side process cannot reliably preserve the workload result, coordinate final metrics, control shutdown, or
remain alive for a bounded final scrape while retaining responsibility for the workload outcome.

### Let MetricShell restart the workload

**Rejected by this decision boundary.**

MetricShell owns one workload execution. Restart behavior is delegated to Docker, Kubernetes, or another orchestrator.
This avoids hidden counter resets, ambiguous final state, and duplicated restart policy.

## Decision boundaries

This ADR does **not** decide:

- the complete workload lifecycle state machine;
- whether process groups are enabled by default for every workload;
- the production workload shutdown timeout;
- the MetricShell shutdown reserve;
- how an external termination deadline is supplied;
- the exact bounded post-exit policy;
- whether exit is controlled by time, scrape count, or both;
- final-scrape eligibility and counting rules;
- Kubernetes Job and CronJob target-discovery behavior;
- production metric-ingestion transports;
- the permanent registry of MetricShell-owned internal exit codes.

These questions remain assigned to subsequent architecture investigations and ADRs, including INV-002, INV-003,
INV-011, and INV-012.

## Implementation requirements

Production implementation derived from this ADR must include automated tests for:

- MetricShell running as PID 1;
- MetricShell running below Docker `--init`;
- signal forwarding to a direct child;
- signal forwarding to a workload process group;
- shell parent and grandchild signal receipt;
- exit codes `0` and arbitrary non-zero workload codes;
- signal-based exits including `SIGTERM` and `SIGKILL`;
- workload start failure;
- MetricShell internal failure;
- non-cooperative workload force termination;
- high child churn;
- orphan descendant adoption and reaping;
- external init with and without subreaper mode;
- bounded post-workload availability with preserved result;
- race-free startup readiness before signal-driven test actions.

The research runner and committed evidence are architecture evidence. Production code must have independent unit,
integration, and end-to-end coverage.
